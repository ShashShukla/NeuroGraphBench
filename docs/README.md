# NeuroGraphBench — user guide

Single-page guide complementing the top-level [`README`](../README.md) and [`ARCHITECTURE.md`](../ARCHITECTURE.md). Use this for "how do I actually use it" questions; use `ARCHITECTURE.md` for design rationale and the resource map.

## Mental model

NGB exposes a fly-brain connectome as a set of *resources* (neuropils, neurons, celltypes, synapses, columns) accessed through:

1. A **REST API** (`fba.server`) that wraps a NeuroArch (neo4j) instance.
2. A **Python client** (`fba.client`) that hides HTTP/JSON and gives you either:
   - **raw methods** — `client.getNeuropilMesh(name)` returns a `navis.Volume`,
   - or an **object model** — `client.neuropil(name)` returns a `Region` with lazy `.mesh`, `.neurons`, `.celltypes`, `.subregions`, `.synapseTable()`, `.connectivity(...)`, `.arborization(...)`.

Both surfaces share the same de-serialization layer, so they return the same Python types.

## Installation

The release is tested on Python 3.10. Use a dedicated conda env to isolate from older fly-brain environments (e.g., `ffbo`, which pins `pydantic 1.x` for `spacy` / `datapane` compatibility).

```bash
conda create -n ngb -y python=3.10
conda install -n ngb -y -c conda-forge \
    h5py numpy pandas scipy networkx pytest httpx requests plotly seaborn
conda run -n ngb pip install -e .
```

**Why install `h5py` from conda-forge first?** `navis` pulls `h5py` in, and on hosts with HDF5 < 1.10.7 the pip wheel build fails. The conda-forge wheel ships its own HDF5, sidestepping the system library entirely. After `h5py` is on the env, `pip install -e .` resolves cleanly.

## Running the FBA server

```bash
conda run -n ngb uvicorn fba.server.app:app --reload
```

Connection settings (neo4j URI, user, password, database) are currently hardcoded at `fba/server/neo4j_driver.py:14-17`. Replace those literals to point at your NeuroArch instance; the config-loading refactor is a deferred TODO.

Useful interactive surfaces:

| URL | What it gives you |
| --- | --- |
| `http://localhost:8000/docs` | Swagger UI with every endpoint, request body editor, and "try it out" buttons. Generated by FastAPI from the route signatures and Pydantic schemas. |
| `http://localhost:8000/redoc` | ReDoc rendering of the same schema. Better for reading; no "try it" button. |
| `http://localhost:8000/health` | Liveness check (`{"status": "ok"}`). |

## Key concepts

**`Region`** — a neuropil, subregion, or arbitrary volume. The single most-used object. Construct via `client.neuropil(name)` (named), `client.region(volume=mesh)` (arbitrary), or `client.region(neuropil=name, volume=mesh)` (LPU — named neuropil restricted to a sub-volume).

```python
lop = client.neuropil("LOP")
lop.mesh                              # navis.Volume (lazy)
lop.subregions                        # list[Region], one per LOP layer
lop.neurons                           # list[str]
lop.celltypes                         # list[str]
lop.synapseTable()                    # pandas.DataFrame
lop.connectivity(entity="celltype",
                 layout="matrix")     # pandas.DataFrame (pivoted)
lop.arborization(names=lplc2_names)   # pandas.DataFrame
```

**`Client`** — holds the base URL and HTTP session, plus the raw-method API.

```python
fba = Client("http://localhost:8000")
fba.listNeuropils()                     # list[str]
fba.getConnectivity(names=["A","B"],
                    region=lop,
                    layout="matrix")    # pandas.DataFrame
fba.getInnervation(names=names,
                   regions=layer_meshes)
```

**Wire types** (Pydantic models in `fba.server.schemas`) — what crosses the wire:

| Type | Shape | Hydrates to |
| --- | --- | --- |
| `MeshPayload` | `{name, vertices: [[x,y,z]...], faces: [[i,j,k]...]}` | `navis.Volume` |
| `TablePayload` | `{columns: [...], data: {col: [vals]}}` | `pandas.DataFrame` |
| `RegionSpec` | `{neuropil: str?, volume: MeshPayload?}` | client-side `Region` |
| `SynapseCloudPayload` | `{locations: [[x,y,z]...]}` | `numpy.ndarray` |

## Common workflows

**Visualize a region's mesh:**

```python
import plotly.graph_objects as go
m = client.neuropil("LOP").mesh
go.Figure(go.Mesh3d(
    x=m.vertices[:,0], y=m.vertices[:,1], z=m.vertices[:,2],
    i=m.faces[:,0], j=m.faces[:,1], k=m.faces[:,2],
    opacity=0.3,
)).show()
```

**Per-layer arborization for a circuit:**

```python
lop = client.neuropil("LOP")
lplc2 = [n for n in lop.neurons if n.startswith("LPLC2_")]
per_layer = {
    layer.name: layer.arborization(names=lplc2, entity="neuron")
                     .set_index("name")["inputs"]
    for layer in lop.subregions
}
import pandas as pd
pd.DataFrame(per_layer).fillna(0).astype(int)
```

The reference notebook [`examples/01-regional-analysis.ipynb`](../examples/01-regional-analysis.ipynb) walks the LPLC2 example end-to-end with visualizations.

## Running the tests

```bash
conda run -n ngb pytest
```

57 tests across server endpoints, Cypher dispatch helpers, client deserialization, and the `Region` object model. The neo4j driver is mocked via `app.dependency_overrides[getDriver]`, so no live database is required.

To add a new endpoint, the typical sequence is:

1. Add Cypher + helpers in `fba/server/cypher/<resource>.py`.
2. Add Pydantic request/response schemas in `fba/server/schemas.py` if needed.
3. Wire the route in `fba/server/routers/<resource>.py`.
4. Add a raw `Client` method in `fba/client/http.py` + de-serializer in `fba/client/deserialize.py`.
5. Surface it on `Region` (or the next object class) in `fba/client/models.py` if it fits the relational view.
6. Tests: unit-test the Cypher helper, integration-test the HTTP path with mocked driver, client-test the de-serialization.

## Naming conventions

- **Function / method names:** camelCase (`getNeuronSynapseTable`, `arborizesIn`). Matches the existing FBA codebase and all repo-root notebooks.
- **Class names:** PascalCase (`Region`, `Neuron`, `Client`).
- **Local variables and parameters:** snake_case (`neuron_name`, `region_table`).
- **JSON wire keys & pandas column names:** snake_case (`pre_name`, `post_celltype`).
